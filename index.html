FileServer Quick Start Guide

How do I access the server?

There are a number of ways to connect to the server, each with their own set of pros and cons. For most users’ needs I would think that File Station, the built-in file manager that you can access from anywhere with your web browser, is probably the way to go. File Station provides a web-based file manager similar to Dropbox, Google Drive, or Microsoft OneDrive. You can drag and drop files to upload them and move them around. If you’re looking to batch process lots of files at once, especially if to multiple locations on the server, you may want to use one of the other options which are covered in the following sections. Or maybe you just prefer to use something like FTP . The bottom line is that you should use what works best for you and for what task you are working on.

How to Mount FileServer as Network Volume Mac only

Sometimes it may be more useful to mount the FileServer as a network shared volume, such as when copying large quantities of files to/from the server. Whether on a Mac or Windows PC, this will make the server appear as just another folder in Finder (Mac) or File Explorer (Windows) allowing drag ’n drop actions involving one or multiple files selected at once.

Windows

  1. Open File Explorer and select This PC
  2. Click the ‘Map network drive’ button in the ribbon menu at the top, then select “Map network drive.” (This is under the Computer tab and should open automatically when you go to This PC) 3.~~ In the Drive list, select a drive letter. (Any available letter will do.), then ensure you have checked the box for “Connect using different credentials”~~

  3. In the Folder box, type the address of the server using the following format:

    ~~\\msgfileserver.myqnapcloud.com\fileserver~~

Windows computers use the SMB/Samba protocol to connect to shared volumes which is not safe to impement over the Internet so this has been blocked. However, you can still connect in this way when using an encrypted VPN connection. The server supports OpenVPN and PPTP type VPN connections, including the one built-in to myQNAP Connect software. If you would like to set this up please send me a note.

Mac

  1. From Finder, go to "Go" > "Connect to Server" to establish a connection with the QNAP NAS via SMB or AFP**. Alternatively, you can use the shortcut keys “Command + K” on your keyboard.

  2. Enter the server address like this:

    afp://msgfileserver.myqnapcloud.com

    Enter your username and password when prompted, selecting “Log in as registered user…”.

FTP

The File Transfer Protocol has been around for a long long time and provides a very stable, fast, and secure connection. If anyone really prefers to connect using an FTP client, it can be done and I would be happy to assist if you send me a request via email. The server supports secure connections only using FTPS (FTP EnforcedSSL) I will be posting updated instructions for using FTP over an encrypted SSL connection in the next couple of weeks. In short, use the server address (msgfileserver.myqnapcloud.com), standard port # 21, mode = passive, must use SSL. Different ftp clients call this different things sometimes, but FTP with SSL, FTPS, FTPES, FTP Explicit Auth, are all examples. SFTP is not the same and will not work.

File Server Administration

Just a few quick notes about this because it’s way too much information to type out here. The gist of it is that the file server and the network it is on is safe and secure. There are literally thousands of attempts to hack the server every day, many of which come from other countries like china and easter block old soviet countries. Some are US though too. The main security against this is monitoring the IP addresses and banning the associated ip address after x failed attempts in x minutes for x long. For SSH, the most dangerous and by far the most tried, this is 5 attempts in 5 mins = lifetime permanent ban. For FTP, AFP and SMB it is much more relaxed at 10 attempts in 15 mins = 1 day ban.

Periodic maintenance includes rotating log files, cleaning the filesystem, and anti-malware and virus scans. keeping software up-to-date is also important. Checking log files for suspicious activity is a must. The server sends all of its logs to my company’s syslog server in the cloud over a secure connection.

Data Backups

The RAID5 array provides some protection in the form of being able to survive the loss of a single hard disk and not lose any data. This is good but should not be thought of as a backup of any kind because it is not. There is only one copy of the data. Data is backed up in 2 additional places, so there are 3 copies of the data at all times.

The first backup is a local backup to an external hard drive that is attached to the server. This provides for fast and easy access to said data, but it also means that in the event of catastrophic failure, both the backup and original data will be lost, due to the fact that they share the same physical space (eg. thief would take everything and fire would destroy everything).

The 3rd and safest backup involves saving the entire contents of fileserver in the cloud. This is cost effective and provides the safest possible means for backup. We send nightly backups to AWS S3 buckets in the us-west-2 region (oregon), to save money over N. California. Data is stored there, in Standard Infrequent access, for 30 days. After 30 days it is archived to ultra low-cost Amazon Glacier storage, which provides excellent long-term storage for the absolute lowest price. The downside is that it may take several days to get the data back out in the event of needing to restore, but we’re only talking about 1-3 days at most.
